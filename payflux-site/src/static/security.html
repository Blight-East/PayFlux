Security

How PayFlux protects your data, secures infrastructure, and maintains the trust required for payment risk operations.

Data Handling and Privacy

PayFlux is designed around a principle of data minimization. The platform processes hashed identifiers and bucketed values — never raw PII, card numbers, or account credentials.

  • Hashed merchant IDs — SHA-256 hashed before ingestion; original values never stored
  • No raw PII — PayFlux does not process cardholder names, addresses, or card numbers
  • Bucketed amounts — Transaction amounts classified into ranges, not stored as exact values
  • Aggregation over extraction — Risk signals are derived from patterns, not individual transaction details

Encryption

All data is encrypted in transit and at rest using industry-standard protocols.

  • In transit — TLS 1.3 enforced on all API endpoints and dashboard connections
  • At rest — AES-256 encryption for persisted data stores
  • In processing — Redis streams use authenticated connections with automatic stream trimming to limit data retention

Authentication and Access Control

PayFlux enforces strict authentication and tenant isolation at every layer of the stack.

  • API authentication — Bearer token authentication with scoped API keys, rotation support, and revocation
  • Dashboard authentication — Clerk-based authentication with multi-factor support
  • Tenant isolation — Row-level security ensures merchants can only access their own data
  • Debug endpoints — Gated to development environments only; inaccessible in production

Infrastructure Security

PayFlux infrastructure follows defense-in-depth principles with multiple layers of protection.

  • Network isolation — Backend services run in isolated network segments with minimal public exposure
  • Dependency management — Automated vulnerability scanning on all dependencies
  • Secret management — Environment-based secret injection; no secrets in source code or build artifacts
  • Monitoring — Real-time alerting on authentication failures, rate limit breaches, and anomalous access patterns
